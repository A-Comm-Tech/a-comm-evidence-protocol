Open Protocol Specification

A-Comm Evidence Protocol (AEP) v1.0.3-rc.2 (Draft)

A cryptographically verifiable evidence standard for agentic commerce transactions — from AI-driven discovery through payment authorization and fulfillment.

v1.0.3-rc.2 (Draft) Apache License 2.0 Published by A-Comm Technologies, Inc. 2026-07-02

Abstract

The A-Comm Evidence Protocol (AEP) defines a standard for generating, storing, and verifying cryptographic evidence of agentic commerce transactions. As AI agents increasingly drive product discovery and purchase decisions, merchants and payment processors require a dispute-grade audit trail that can demonstrate the provenance of any transaction — from the AI platform that surfaced the product through to payment authorization and fulfillment.

AEP specifies a sequential hash chain of typed artifacts, each immutably linked to its predecessor via SHA-256. The chain spans eight defined layers: discovery, referral, intent, delegation, policy, cart, authorization, and fulfillment. Any insertion, deletion, or modification of an artifact is cryptographically detectable at every subsequent node in the chain. The protocol is designed to be implementable by any PSP, commerce platform, or merchant independently of A-Comm Technologies, Inc.'s infrastructure, under Apache License 2.0 with explicit Apache §3 patent grant on all claims A-Comm Technologies, Inc. holds in the Necessary Claims for this specification.

v1.0.3 extends the §3.9 delegation wrappers to cover Skyfire KYA (agent identity, trust/identity-transport layer) and Skyfire KYA Pay (identity-linked payment, settlement layer), and the Web Bot Auth wire-identity envelope (Cloudflare / IETF draft-meunier-web-bot-auth-architecture). It adds an optional request_binding primitive — a witness-side observation record binding a bearer credential to the specific HTTP request it accompanied — and promotes non-card settlement (stablecoin / account / real-time rails) to first-class in §3.6. The cross-protocol witness now spans every shipping agentic-commerce trust and payment artifact known as of 2026-06-29.

1. Introduction

1.1 Problem Statement

AI-mediated commerce is growing rapidly. Consumers increasingly discover and purchase products through AI chat interfaces where product recommendations drive real transactions. This creates a new class of dispute risk:

  • The consumer interacted with an AI agent, not a human or traditional web UI
  • No browser-native referrer chain is reliable across AI platforms
  • The authorization trail exists, but the intent trail — what was asked, what was recommended, what was confirmed — is typically absent from PSP evidence
  • Merchants cannot prove to card networks that a consumer explicitly authorized an agentic transaction

AEP solves this by defining a tamper-evident evidence chain that captures every step from AI-driven discovery through order fulfillment, producing a dispute-grade artifact bundle that merchants can submit to PSPs and card network reviewers.

Since v1.0.1 was published, the agentic commerce ecosystem has consolidated around three signature envelope families. SD-JWT-VC (RFC 9901) is now the canonical mandate envelope for Google AP2 v0.2; Mastercard Verifiable Intent v0.1-draft uses layered SD-JWT / KB-SD-JWT envelopes, with VC integration on its roadmap. RFC 9421 HTTP Message Signatures is the canonical per-request identity envelope across Visa TAP, Cloudflare Web Bot Auth, and UCP request/response signing. JWS (RFC 7515) persists at higher payload layers (Visa Intelligent Commerce Agentic Consumer Recognition + Payment Container; A2A signed Agent Cards using RFC 8785 JCS canonicalization). AEP does not generalize one wrapper across all three families — they have different verification procedures and different evidence semantics — but instead records which envelope was used per protocol sub-wrapper.

Three independent specs (AP2 v0.2, UCP 2026-04-08, Mastercard VI v0.1) converged on the same cart↔payment hash-binding construct between February and June 2026: transaction_id == base64url(sha256(checkout_jwt)), with the cart-confirmation hash equivalent across the chain. This is now the universal cart-↔-payment integrity primitive in agentic commerce. AEP §3.5 and §3.9.11 rule 4 are updated in v1.0.2 to use cryptographic hash equivalence rather than the v1.0.1 semantic items-and-amount matching.

1.2 Design Goals

  • Tamper-evident: Any modification to any artifact in the chain is detectable at all subsequent nodes without requiring a trusted third party.
  • Dispute-grade: The evidence bundle maps directly to PSP and card-network representment evidence fields.
  • PII-minimal: Consumer identifiers are pseudonymized. No raw card data, no PAN, no personal identifiers in any artifact payload.
  • Platform-agnostic: Works across any commerce platform and any PSP.
  • Open: Apache 2.0. Any implementation can generate and verify AEP chains without dependency on A-Comm Technologies, Inc.
  • Attribution-aware: Records the inbound AI referral when a consumer arrives from an AI surface, plus an optional AI-platform citation attestation when the platform itself reports it. AEP asserts no visibility into the AI surface.
  • Cross-protocol witness: An AEP chain is the only evidence artifact that records which protocol(s) carried a given transaction. No competing protocol produces this cross-protocol interoperability record on its own.
  • Open/Closed mandate-aware: Evidence captures the open-mandate vs closed-mandate distinction across AP2 (mandate.checkout.open.1 vs mandate.checkout.1) and Mastercard VI (kb-sd-jwt vs kb-sd-jwt+kb). An autonomous-mode agent-signed-only transaction is materially weaker evidence than a user-signed-at-checkout transaction and is explicitly tier-distinguished in §3.9.12.
  • Cryptographic binding over semantic matching: Cart↔payment integrity is verified by hash equivalence (checkout_hash == transaction_id), not by semantic field comparison. Implementations MUST NOT depend on item-name normalization or amount-range tolerances for binding integrity.

2. Data Model

An AEP commerce journey is modeled as a single transaction chain — an ordered sequence of immutable artifacts. Each artifact records one event in the consumer's journey and is cryptographically linked to its predecessor.

2.1 Commerce Journey Layers

A complete agentic commerce transaction spans up to eight artifact layers. The first two (discovery and referral) are present only when the consumer arrived via an external AI platform; the delegation layer (inserted at position 4 in v1.0.2) is present whenever an agentic protocol is in play:

1
Discovery ai_referral types only

An optional, platform-attested record of a product citation inside an AI surface. It can be populated only when the AI platform itself reports the citation — AEP has no visibility into what happens inside an AI chat surface, so a merchant cannot observe or reconstruct it, and most chains carry no Discovery artifact. The observable inbound signal is the Referral artifact (§3.2). Records the cited product URL and the attribution method used to link the AI referral to the inbound session; it never records the consumer's raw query.

  • Actor type: agent (AI platform, self-reported) — populated only when the platform attests the citation
  • Key payload fields: platform, product_url_cited, attribution_method, attribution_confidence, citation_position (optional) — no raw query
2
Referral ai_referral types only

Records the inbound session that originated from the AI citation — the click from the AI platform to the merchant's site, including session token, referral event ID, and the product that was cited.

3
Intent

Records the consumer's (or their agent's) explicit purchase intent: the product selected, quantity, variant, and the actor responsible. This is the earliest layer present for all transaction types.

  • Key payload fields: product_id, product_name, quantity, variant, query
4
Delegation agentic protocols only

Records the handshake between the agent and the merchant that authorizes the transaction — agent identity, consumer authorization, merchant acceptance, and protocol provenance. Present whenever an agentic protocol is in play; omitted when protocol = GENERIC. See §3.9.

  • Actor types: agent, merchant, credentials_provider
  • Key payload field: protocol_metadata (§3.9.4)
5
Policy

Records the outcome of the merchant's policy engine evaluation — whether the transaction was permitted or blocked, which rules were evaluated, and the evaluation result. Required on all transaction types. The policy check must occur before authorization.

6
Cart

Records the consumer's explicit cart confirmation before checkout — the items in the cart, total amount, currency, and the confirmation signal. Provides the "authorization trail" for the agentic transaction.

  • Key payload fields: confirmed (boolean), items (array), total_amount, currency
7
Authorization Seal Point

Records the PSP payment authorization result. The current_hash of this artifact becomes the sealed_bundle_hash — the immutable seal of the transaction. Post-seal, only fulfillment artifacts may be appended.

  • Actor type: psp
  • Key payload fields: payment_method_token (never raw card data), psp, authorization_code, amount_authorized, currency
8
Fulfillment

Records the fulfillment provider's response: carrier, tracking number, shipping status, and delivery confirmation. May arrive asynchronously after the chain is sealed. Implementors must support a late-arrival append path that does not require breaking the seal.

  • Actor type: fulfillment_provider
  • Key payload fields: carrier, tracking_number, status, shipped_at, delivered_at
ⓘ
Editorial note — layer count v1.0.1 described seven layers and listed delegation only inside §3.9 prose; the artifact-type enum in Appendix B omitted it. v1.0.2 promotes delegation to a first-class layer in the journey.

2.3 Sequential Hash Chain Model

AEP uses a sequential previous-hash model where previous_hash is included inside the hash input of every artifact. This is distinct from reference-pointer hash chain models:

Model Formula Chain break detection
AEP (sequential) current_hash = SHA256(canonical({...all_fields, previous_hash})) Any insertion, deletion, or mutation is detectable at every subsequent artifact
Reference-pointer model current_hash = SHA256(payload) + separate chained_from pointer Only detects breaks if the pointer is verified separately — pointer can be silently remapped

Because previous_hash is inside the hash input, any reordering or insertion in the sequence invalidates the hash of every subsequent artifact. Verification does not require a trusted third party — anyone with the artifact sequence can independently verify integrity.

The property of previous_hash being included inside the hash input is a MUST under §0.1 Conformance. Implementations that compute current_hash without previous_hash in the input are non-conformant and produce an undetectable chain-break vulnerability.

2.5 Sealing Model

The chain is sealed at the authorization layer. Sealing means:

  • The current_hash of the authorization artifact is stored as sealed_bundle_hash on the transaction chain record
  • sealed_bundle_hash is immutable once set — no update path exists
  • Only post-seal artifact types — fulfillment and refund (§3.10), plus annotation/rebuttal records per §6.5 — may be appended post-seal, via the async late-arrival append path
  • The fulfillment artifact includes the sealed authorization hash in its chain, so any post-seal fulfillment record is still cryptographically bound to the sealed chain

The delegation artifact, when present, is pre-seal. Its current_hash is bound into the chain at position 4 and forms part of the input to authorization's previous_hash (via the intermediate policy and cart artifacts). The seal point remains at authorization.

⚠
Late-arrival fulfillment Fulfillment artifacts often arrive asynchronously after the chain is sealed (shipping events, delivery confirmations). Implementors must support appending fulfillment artifacts post-seal. The sealed_bundle_hash must not be invalidated by post-seal fulfillment appends.

2.6 Protocol Interoperability

AEP is protocol-agnostic. It does not replace or compete with the emerging agentic commerce protocols — it wraps them. Every AEP artifact carries a protocol field identifying which protocol carried the handshake it represents, and a protocol_metadata sub-object carrying the protocol-specific payload (signatures, mandates, session IDs, tool call records, settlement proofs).

Layer-Aware Composition

Layer membership is normative. When wrappers from different layers (e.g., A2A as identity-transport carrying AP2 as commerce-delegation) co-exist, all SHOULD be populated. The intra-layer orderings below are informative: they are selection guidance for the protocol field when multiple wrappers within a layer are populated, stating which wrapper best identifies the transaction's commerce semantics. They imply no quality ranking of the protocols. (This framing replaces the v1.0.1 flat dominance ordering and, in rc.2, the v1.0.2/v1.0.3 normative per-layer dominance orderings.)

  • commerce-delegation layer: wrappers are selected in the order listed — AP2, MASTERCARD_VI, UCP, ACP (informative)
  • identity-transport layer: wrappers are selected in the order listed — TAP or SKYFIRE_KYA, then WEB_BOT_AUTH, then A2A (informative; updated in v1.0.3). Selection rationale: TAP and Skyfire KYA carry a vetted/issued agent identity; Web Bot Auth proves agent legitimacy on the wire but carries no human/business binding; A2A is a carrier.
  • settlement layer: x402 and SKYFIRE_KYA via its kya_payment facet (peers to commerce-delegation, not subordinate; Skyfire settlement joined in v1.0.3)
  • tool-invocation layer: MCP (peer to commerce-delegation, not subordinate)
  • discovery layer: discovery_source block on the §3.1 Discovery artifact (NOT a §3.9 sub-wrapper)
  • fallback: GENERIC

Multi-protocol chains. A single transaction MAY (and often does) carry multiple protocols simultaneously. The strongest evidence configurations combine wrappers from multiple layers: TAP (identity-transport) + AP2 (commerce-delegation) + x402 (settlement) is the S-tier-eligible composition. Implementations SHOULD capture every protocol envelope observed during the transaction.

Fallback behavior. Transactions that occur without any standard agentic protocol in play (for example, a consumer completing a traditional web checkout with no agent involvement) MUST set protocol to GENERIC and MUST omit the delegation artifact entirely. In this case, the chain proceeds directly from intent to policy.

2.7 Cross-Artifact Required Fields

The following fields MUST appear on every artifact in every chain, regardless of type. They establish chain identity, scoping, and deterministic ordering. Implementations MAY carry the chain-level fields (chain_id, merchant_id, session_id_hash, spec_version) at the bundle level rather than duplicating them per artifact. The chain-level fields MUST appear on every artifact record (or at the bundle level) and are integrity-protected by the chain seal (§2.5) and the bundle signature (§4.3) — but they are NOT part of the per-artifact hash input, which is defined solely by §4.1 and Appendix B.

Field Type Description Scope
chain_id string (UUID) Globally unique identifier for the transaction chain this artifact belongs to. Chain-level
merchant_id string The merchant whose chain this is. All verification MUST scope queries by this value — a bundle referencing a merchant_id the verifier does not recognize MUST be rejected. Chain-level

session_id_hash Derivation

The session_id_hash HMAC derivation is defined explicitly (was prose-only in v1.0.1):

session_id_hash = "sha256:" + lowercase_hex(
  HMAC-SHA256(
    key     = per-merchant-server-side-salt,
    message = utf8(session_token)
  )
)

Salt rotation policy. The per-merchant salt MUST be rotated at most once per 365 days. Rotation MUST NOT invalidate existing chains: the old salt MUST remain available for verification of chains generated under it. Implementers MUST store a salt_kid reference per chain and a salt-key map indexed by salt_kid in a key-management system. The reference implementation's salt_kid field is added at the chain level per this rule.

⚠
Secrets never in payloads Session tokens, auth cookies, API keys, PSP secret keys, and any equivalent credential MUST NEVER appear in an artifact payload. Evidence bundles are designed to be exportable to third parties; any secret in a payload becomes abuse-capable on export. Use session_id_hash and payment_method_token-style references instead.

2.8 Signal Classification

Every field in every artifact payload has a signal class that tells dispute reviewers, regulators, and downstream systems how much weight to place on the field in evidentiary use. Signal classification is a first-class AEP concept and MUST be preserved on export. The four classes are:

Class Meaning Examples Verification class
Deterministic Verifiable by independent re-computation, returned by a trusted system (PSP webhook, platform webhook, signed API), or computed from two deterministic inputs. network_transaction_id, charge_id, auth_code, avs_result, delivery_address_hash. Verifiable by independent recomputation.
Attested Cryptographically signed by a party with standing — user (via WebAuthn or AP2 VDC), merchant (via JWKS-published key), PSP, agent (via TAP). AP2 ap2_user_signature, TAP tap_signature, merchant session_binding co-signature; risk_score / fraud_score (model outputs attested by their producer — reclassified from Deterministic in rc.2). Verifiable against the signer's published key.
Inferred Reconstructed from observable signals (referrer headers, URL parameters, behavioral heuristics) rather than proven. Best-effort attribution. Discovery platform (inferred from referrer/params), attribution_confidence, consumer_ip_country from IP geolocation. Corroborated inference — attribution, not proof.
Asserted Unsigned claim by one party (usually the merchant) with no independent corroboration. Weakest signal; included for completeness but MUST NOT be relied on alone. Discovery citation_position (only available if agent reports it), quantity (unless AP2-signed). Uncorroborated statement.

Classification is per-field, not per-artifact. A single artifact MAY carry fields of multiple classes — for example, the Authorization artifact's network_transaction_id is Deterministic, while fraud_score is Attested (a model output attested by its producer, the PSP; reclassified from Deterministic in rc.2), and hypothetical custom merchant fields would be Asserted. Implementations MUST propagate the signal class through exports (§6) so representment reviewers see it.

Verification class is recipient-relative for HMAC-derived fields. HMAC-derived fields (session_id_hash, consumer_ip_hash, query_hash, address hashes) are Deterministic only for verifiers with access to the relevant §2.7 salt; for any verifier without salt access they carry the Asserted class for that verifier.

Strengthening signals is an implementation goal. A chain with many Attested and Deterministic fields is evidentiarily stronger than one with many Asserted fields. §3.9.12 defines an evidence strength tier (S/A/B/C/D) derived from the Delegation artifact's signal composition; future AEP revisions MAY extend the tier calculation across the full chain.

The evidence-strength tier defined in §3.9.12 is a chain-level aggregate of per-field signal classes. A chain whose Delegation artifact carries many Attested fields (cryptographically signed) tiers higher than one whose Delegation artifact relies on Asserted fields (unsigned merchant claims). Tier computation is deterministic and recomputable from any archived chain.

3. Artifact Types

Each artifact type defines the required and optional payload fields, the permitted actor types, and the position in the commerce journey sequence.

3.1 Discovery Artifact

The discovery artifact is an optional, platform-attested record of the moment an AI platform surfaced a merchant product in a consumer-facing response. It can be populated only by the AI platform itself (see the Minimal Platform Profile, §8.7): AEP has no visibility into an AI chat surface, so this artifact cannot be observed or reconstructed by a merchant and is absent from most chains. The consumer-observable signal — arrival from an AI surface — is recorded by the Referral artifact (§3.2), not here.

✓
What Discovery adds — only when the platform participates When an AI platform attests a citation, a reviewer can trace an AI-mediated transaction back to the platform and product citation that originated the journey. Because this attestation is optional and platform-supplied, the durable, always-present attribution signal is the Referral artifact (§3.2); Discovery enriches it only when the platform participates.

3.5 Cart Artifact

Records the consumer's explicit cart confirmation — the final review step before checkout. The confirmed: true field plus the matching AP2 Cart Mandate signature (carried in §3.9 Delegation) are the dispute-resolution signals that demonstrate the consumer (or their agent acting with consent) explicitly approved the cart contents before payment.

⚠
Addresses must be hashed, not stored raw Shipping and billing addresses are PII and MUST NOT appear in payload fields as raw strings. Use sha256(canonical_address_string + per-merchant-salt). The hash enables address-match verification against the fulfillment artifact (§3.7 delivery_address_hash) and the authorization artifact's AVS result, without exposing the address itself when the bundle is shared.

Actor type: fulfillment_provider. Signing actors: fulfillment_provider (carrier-attested) + merchant (wrapper). The merchant's server signs the artifact wrapper. delivery_address_match: true is the single most valuable representment signal a chargeback reviewer looks for when the dispute reason is "item not received." Its value is deterministic — computed from two already-present hashes — and cannot be forged without breaking the hash chain upstream.

3.9 Delegation & Protocol Metadata

ⓘ
Normative — introduced v1.0.1; major sub-wrapper alignment in v1.0.2; extended in v1.0.3 The delegation artifact type and its protocol_metadata payload record the handshake between the agent and the merchant that authorizes the transaction. This is where the agentic commerce protocols (TAP, AP2, MASTERCARD_VI, ACP, UCP, MCP, x402, A2A, SKYFIRE_KYA, WEB_BOT_AUTH) are captured in the evidence chain. The §3.9 sub-wrapper schemas, verification rules, and evidence-strength tiers in this version are NOT cross-compatible with v1.0.1; verifiers MUST gate behavior by spec_version per §2.7.

3.9.1 Purpose

The delegation artifact records four claims simultaneously:

  • Agent identity — which agent is operating (TAP-signed when available)
  • Consumer authorization — that the consumer delegated purchase authority to the agent (AP2 user-signed mandate when available)
  • Merchant acceptance — that the merchant accepted the handshake and bound the session (ACP session ID, UCP checkout binding)
  • Protocol provenance — which protocol(s) carried the handshake, for downstream evidence evaluation (§3.9.12)

3.9.2 Chain Position

The delegation artifact sits after intent and before policy. Sequence:

discovery → referral → intent → delegation → policy → cart → authorization → fulfillment
⚠
Strengthened normative rule (replaces v1.0.1 "MAY be omitted") Delegation MUST be omitted when protocol = GENERIC AND no sub-wrapper is populated (traditional web checkout with no agent involvement). Delegation MUST be present when ANY of the following protocols is in play: TAP, AP2, ACP, UCP, MCP, x402, A2A, MASTERCARD_VI. When omitted, the chain proceeds directly from intent to policy.

Delegation is required for transaction types that include an agent session (see §3.9.2).

3.9.3 Permitted Actors

  • agent — primary producer of the delegation artifact
  • merchant — co-signer when merchant acceptance is captured in the same artifact
  • credentials_provider — co-signer when an AP2 Credentials Provider is in the chain

Every delegation artifact MUST be signed by at least one actor whose public key is independently verifiable per §3.9.11 rule 1 via that actor's canonical key-discovery path. Multi-signature delegation (agent + merchant + credentials_provider) is RECOMMENDED when the underlying protocols support it.

3.9.11 Verification Requirements

Verification MAY use third-party key material witnessed at capture time per the §4.3 key-witness snapshot rule (e.g., when an upstream key has rotated since capture); when witnessed rather than live keys are used, that fact MUST be recorded in the verification result.

  1. Signature verification with canonical key discovery. Every non-null sub-wrapper's signatures MUST verify against keys discovered via that sub-wrapper's canonical key-discovery path:
    • TAP: Signature-Agent header → JWKS directory; or tap_directory_url when present
    • AP2: inline cnf.jwk per SD-JWT-VC
    • A2A: a2a_jwks_url for Agent Card signatures
    • UCP: merchant_public_keys_url (JWKS at /.well-known/ucp)
    • ACP: pre-shared HMAC indexed by merchant_id
    • MCP: OAuth iss + kid (RFC 8707 resource binding)
    • x402: facilitator_signer (recovered from EIP-712 signature)
    • MASTERCARD_VI: layered key model per rule 8 (L1: credential provider's published keys; L2/L3: keys bound via the preceding layer's cnf); ES256 only
  2. Timestamp skew. Sub-wrapper timestamps MUST be within 24 hours of the delegation artifact's top-level captured_at. Per-protocol skew rules apply where the protocol specifies one (TAP expires − created ≤ 480 s; AP2 mandate exp rule per SD-JWT-VC; x402 valid_after/valid_before).
  3. TAP authority and expiry. When agent_identity is present, the @authority component covered by the TAP signature MUST match the merchant domain on file for this chain, AND tap_expires − tap_created MUST NOT exceed 480 seconds.
  4. AP2 cryptographic cart-payment binding (replaces v1.0.1 semantic matching). When consumer_authorization is present, ap2_checkout_hash MUST equal base64url(sha-256(ap2_checkout_jwt)) AND the Cart artifact's cart_payment_binding.binding_value MUST equal that hash. Cryptographic equivalence replaces v1.0.1 rule 4's semantic items+amount matching.
  5. Opaque raw payload. The raw_protocol_payload field, when present, is opaque to AEP verification and is preserved for downstream tooling.
  6. ACP SPT cap. When session_binding_acp.delegated_payment_token is present, Cart total ≤ max_amount, Cart currency == token currency, Cart cart_confirmed_at ≤ expires_at.
  7. x402 EIP-712 signer recovery. When http_payment.scheme === "exact" AND http_payment.network starts with "eip155:", verifiers MUST recover the EIP-712 signer from x402_payment_signature and assert that the recovered address equals payer.
  8. Mastercard VI SD-JWT / KB-SD-JWT chain. When mastercard_verifiable_intent is present, the layered credential chain MUST verify per its key model: L1 verifies against the credential provider's published keys; L2 verifies against the user key bound via L1's cnf; L3 verifies against the agent key bound via L2. The sd_hash chain MUST validate, and ES256 MUST be the only accepted alg.
  9. Skyfire KYA (NEW in v1.0.3). When kya_identity is present, verify the kya_token ES256 signature against the JWKS at jwks_url selected by kid; assert aud matches the chain's merchant/seller-agent id and exp is unexpired. Record hid.verifier/verified provenance via signal classification (§2.8) — a Skyfire-self-asserted verified=false is Asserted; an external identity_verifier-attested verified=true is Attested.
  10. Skyfire KYA Pay (NEW in v1.0.3). When kya_payment is present, verify the payment token's ES256 signature (when only pay_token_sha256 is persisted per §3.9.20, signature verification is performed at capture time and its result recorded); assert Cart total ≤ token amt (same cur); the charges[].value entries MUST sum to ≤ amt; each charges[].charged_at MUST be ≤ the token's chargeable-until time — token exp plus the profile's charge grace window — a bound distinct from settlement deferral, which governs only when a charge settles.
  11. Web Bot Auth (NEW in v1.0.3). When web_bot_auth is present, verify the RFC 9421 signature against the key discovered via wba_signature_agent / directory; assert wba_tag contains web-bot-auth, the @authority covered component matches the merchant domain, and wba_expires − wba_created is within policy.
  12. Request binding (when present; NEW in v1.0.3). Recompute and match request_body_sha256 and target_uri against the request the bound_envelope rode on. A mismatch MUST be surfaced as a request_binding_mismatch warning (non-fatal unless the verifier's policy elevates it).

3.9.12 Evidence Strength Ranking (Verification Classes; Informative for Adjudication)

⚠
Scope of the tiers (normative disclaimers) No card network or payment scheme rule assigns dispute outcome, qualification, or liability consequence to any AEP tier. Tiers describe cryptographic verifiability and evidence composition only. The absence of an AEP chain, or a lower tier, implies nothing about a transaction's legitimacy and MUST NOT be presented as adverse evidence.

S-tier (Strongest). Any one of:

  • AP2-Direct (ap2_mode = direct, both Closed mandates verify) + session_binding_ucp OR session_binding_acp + on-chain x402.transaction_hash (finalized)
  • mastercard_verifiable_intent (any mode) + AP2 cross-references verified
  • TAP with BOTH tap_consumer_recognition AND tap_payment_container populated and verified, alongside any commerce-delegation wrapper

A-tier. Any one of:

  • AP2-Autonomous (ap2_mode = autonomous, both Open and Closed mandate pairs verify) + any session_binding_*
  • mastercard_verifiable_intent autonomous mode with all 8 constraint families verified

B-tier. Any one of:

  • agent_coordination (signed A2A Agent Card per JWS + RFC 8785 JCS) + any session_binding_*
  • mastercard_verifiable_intent immediate mode (2-layer credential chain)
  • session_binding_acp + delegated_payment_token verified
  • tool_invocation (MCP) with verified OAuth + canonical-URI binding co-existing with session_binding_*
  • kya_identity (verified) + any session_binding_* (added in v1.0.3)
  • kya_payment (verified) treated as a constrained spend authorization + any commerce-delegation wrapper (added in v1.0.3)

4. Cryptographic Hash Chain

4.1 Hash Formula

The current_hash of every artifact is computed as SHA-256 over the canonical JSON representation of the artifact's hash input. All fields that contribute to the hash are listed below:

current_hash = SHA-256( JCS-canonical-JSON( {
  actor_id,            // normalized actor identifier
  actor_type,          // 'user' | 'agent' | 'merchant' | 'psp' |
                       // 'fulfillment_provider' | 'credentials_provider'
  artifact_type,       // 'discovery' | 'referral' | 'intent' | 'delegation' |
                       // 'policy' | 'cart' | 'authorization' | 'fulfillment'
  interaction_channel, // 'voice' | 'text_chat' | 'voice_and_text' | null
  metadata,            // {protocol, sdk_version, merchant_id, session_id_hash, ...} | null
  payload,             // type-specific payload object
  previous_hash,       // SHA-256 of preceding artifact | null (genesis only)
  timestamp            // ISO 8601 UTC — RFC 3339 'Z' format enforced
} ) )

The eight fields above are the exhaustive per-artifact hash input, matching Appendix B. The §2.7 chain-level fields (chain_id, merchant_id, session_id_hash, spec_version) are NOT part of the hash input; they are integrity-protected by the chain seal and the bundle signature per §2.7.

Normative Canonicalization Rule

Canonical JSON serialization MUST conform to RFC 8785 (JSON Canonicalization Scheme, JCS). All hash inputs in this specification — including artifact hash inputs (this section), A2A Agent Card signing inputs (§3.9.14), AP2 SD-JWT-VC sd_hash chains (§3.9.7), and MCP arguments / result canonicalization (§3.9.9) — MUST use RFC 8785.

The four normative properties this gives are:

  1. Lexicographic key sort by Unicode code point at every nesting level.
  2. RFC 8259 integer and number representation — no leading zeros, no + sign, no exponent for integers, no negative zero.
  3. NFC Unicode normalization MUST be performed on string keys and values before serialization.
  4. No insignificant whitespace — JCS produces a minimal byte stream.

Integer Constraint

Hash-input numbers MUST be integers — no fractional, no scientific notation, no negative zero. AEP minor-unit amounts are integers per §3.5 and §3.6. The total, subtotal, shipping, tax, discounts, items[*].unit_price, and amount fields are integer JSON values, NOT strings.

⚠
previous_hash must be inside the input The previous_hash field must be included inside the hash input object — not treated as a side reference. This is what makes the chain sequential and tamper-evident. Any implementation that computes the hash without previous_hash in the input is non-conformant and produces an undetectable chain break vulnerability.

Hash Output Format

The hash is output as a hex-prefixed string: sha256:<64-character lowercase hex>.

sha256:a3f1c8b0e2d4f6a1b3c5e7f9a1b3c5e7f9a1b3c5e7f9a1b3c5e7f9a1b3c5e7f9
⚠
Ed25519 signature input encoding The Ed25519 signature input is the 32 raw bytes of the SHA-256 digest — NOT the sha256: prefix + 64-hex-char string form. The string form is for storage, display, and chain-linkage; the raw 32 bytes are the cryptographic input to Ed25519. Implementations MUST sign the raw 32 bytes, not the string form; chains created under either input form remain verifiable under their declared spec_version.

4.2 Genesis Artifact

The first artifact in any chain is the genesis artifact. Its distinguishing property is that previous_hash is null. All subsequent artifacts must have a non-null previous_hash equal to the current_hash of the immediately preceding artifact.

Property Genesis (seq = 1) Non-genesis (seq > 1)
sequence_number 1 N (monotonically increasing)
previous_hash null Required — current_hash of artifact at seq N-1
Chain link verification N/A — no predecessor Must verify previous_hash == preceding.current_hash

4.3 Ed25519 Server Signature

Every artifact carries a required Ed25519 server signature over its current_hash. The signature is stored as a separate field (server_signature) on the artifact record and is never included in the hash input itself.

server_signature = Ed25519Sign(private_key, sha256_raw_bytes)

where sha256_raw_bytes is the 32-byte SHA-256 digest (NOT the sha256: prefix + hex form — see the §4.1 signature input encoding rule).

The corresponding Ed25519 public key is published by the implementing party (A-Comm Technologies, Inc. or any independent implementor) to allow third-party verification without access to the private key.

Key publication and rotation. Implementations MUST publish their Ed25519 verification key at a stable URL (/.well-known/aep-public-key or equivalent), tagged with a key_id. Key rotation MUST include an overlap window of at least 90 days during which both old and new keys are published; chains carry signing_key_id at the chain level so verifiers can resolve the correct key.

Key-compromise revocation (NEW in rc.2). If a signing key is compromised, the implementation MUST publish a revocation statement at the same well-known key URL, identifying the revoked key_id and the revocation time. Verifiers MUST check for a revocation statement when resolving a key; a signature attributed to a time after the stated revocation time MUST fail verification.

External Existence Anchoring NEW in rc.2

The chain and its signature are produced by the implementing party; they attest to integrity, not to existence at a point in time as against that party. Implementations are RECOMMENDED to externally anchor the sealed chain hash (or the exported bundle hash) using at least one of:

  • an RFC 3161 timestamp authority token over the sealed hash;
  • an append-only transparency log entry recording the sealed hash; or
  • a counter-signature from an independent party (an Evidence Custodian per §8.2.1, a PSP, or another non-exporting actor).

A bundle whose seal is not externally anchored, and whose verification was performed only by the exporting party, carries the Asserted verification class as a whole (§2.8, §5); an anchored seal with verification by a party other than the exporter carries a higher class.

Key-Witness Snapshots NEW in rc.2

Disputes arrive months after capture, by which time upstream keys may have rotated. At capture time, implementations SHOULD seal a snapshot of the third-party key material used for envelope verification — JWKS documents, signature-directory entries — into the chain, so that §3.9 envelopes remain verifiable after upstream rotation. Verification against witnessed keys is permitted per the §3.9.11 preamble; the use of witnessed rather than live keys MUST be recorded in the verification result.

🚫
Signature is always required Unsigned artifacts are non-conformant. An AEP dispute bundle that contains artifacts without server signatures cannot be presented as a complete evidence chain. Verification implementations must treat missing signatures as a verification failure.

Verification Steps (per artifact)

  1. Recompute current_hash from the artifact's hash input fields
  2. Assert recomputed hash equals stored current_hash (constant-time comparison)
  3. Verify Ed25519 server_signature against the stored current_hash using the published public key
  4. Verify previous_hash equals current_hash of the preceding artifact (except genesis)

6.2 Chain Re-verification on Export

Before generating a dispute bundle for a representment-bound export, implementations MUST re-verify the full chain. Any verification failure MUST be surfaced in the bundle's verification object rather than silently omitted. A bundle whose verification.valid = false MUST NOT be exported for submission.

Implementations MAY verify at seal time and serve the cached result — carrying a verified_at timestamp — for inquiry-speed channels; full re-verification remains REQUIRED for representment-bound exports. Implementations MUST index bundles at seal time: settlement and fulfillment records are late-arriving enrichments to an already-indexed bundle, not preconditions for indexing.

A bundle's verification result also carries a verification class per §5 and §4.3: a self-verified, unanchored bundle is Asserted as a whole; an externally anchored, independently verified bundle carries a higher class.

Compelling-Evidence Renderer Profile (fraud reason codes)

Pre-dispute inquiry channels expect transaction data to be queryable within roughly 24 hours of authorization and answered at sub-second latency across a 13-month history. 6.5 Named-Actor Rights on Export NEW in rc.2

The annotation artifact's payload schema is scheduled for v1.1.0; annotation artifacts do not count toward the §2.2 expected counts.

Platform attribution carried at the Inferred or Asserted class (§2.8) MUST be labeled "not verified by the named platform" on export, and MUST NOT be presented as deterministic fact absent a countersignature from that platform.

Safe harbor (conformance term). Artifacts emitted by a platform under this standard MUST NOT be presented as evidence against the emitting platform in disputes between that platform and merchants or consumers. Conformant use of this specification is conditioned on this term.

7.3 Session Linkage Chain

Connecting a discovery artifact to a payment authorization requires the ehc_transaction_id to propagate through the consumer's browser session. The reference implementation uses the following linkage chain:

1.  Consumer arrives via AI citation
        → POST /v1/events/referral (referral tracking script)
        → EHC transaction created; discovery + referral artifacts appended
        → Response includes: { ehc_transaction_id: "txn_..." }
        → Store the ehc_transaction_id in the browser session
                 (session storage and/or a Secure, SameSite=None cookie)

2.  Consumer adds product to cart
        → the commerce platform's client-side cart event fires
        → POST /v1/ehc/artifacts  { artifact_type: "intent", ... }
          Auth: session-token header (not cookie)
        → Intent artifact appended (seq 3)

3.  Consumer proceeds to checkout
        → the platform's checkout extension point
        → reads cart lines + total → cart artifact payload
        → POST /v1/ehc/artifacts  { artifact_type: "cart", confirmed: true, ... }
        → Cart artifact appended (seq 5 or 3)
        → the ehc_transaction_id is carried on an order-level metadata field

4.  Consumer pays
        → the PSP creates the payment authorization with metadata:
            { ehc_transaction_id: "txn_..." }
        → the payment-success webhook fires
        → Authorization artifact appended + chain sealed

The ehc_transaction_id is the single token that ties all four steps together. Implementations must propagate it from the initial referral response through to the PSP's payment-authorization metadata. Loss of this token at any step results in an incomplete chain.

8. Implementation Notes

8.1 Immutability Enforcement

AEP artifacts must be append-only. The storage layer must enforce immutability at the database level — application-layer enforcement alone is insufficient because it can be bypassed by direct database access or administrative operations.

Recommended PostgreSQL implementation

-- Prevent mutation of immutable fields
CREATE OR REPLACE FUNCTION prevent_ehc_artifact_mutation()
RETURNS TRIGGER AS $$
BEGIN
  RAISE EXCEPTION
    'ehc_artifacts rows are immutable. Attempted update on id=% field=%',
    OLD.id, TG_OP;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER ehc_artifacts_immutability_update
  BEFORE UPDATE ON ehc_artifacts
  FOR EACH ROW EXECUTE FUNCTION prevent_ehc_artifact_mutation();

-- Prevent deletion
CREATE OR REPLACE FUNCTION prevent_ehc_artifact_delete()
RETURNS TRIGGER AS $$
BEGIN
  RAISE EXCEPTION
    'ehc_artifacts rows are immutable and cannot be deleted. id=%', OLD.id;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER ehc_artifacts_immutability_delete
  BEFORE DELETE ON ehc_artifacts
  FOR EACH ROW EXECUTE FUNCTION prevent_ehc_artifact_delete();

Implementors must also ensure that sealed_bundle_hash on the transaction chain record is set once and never updated. Apply a CHECK constraint or trigger on the column to enforce this.

8.2 PII and Privacy

AEP is designed to be PII-minimal. The following rules apply to all artifact payloads:

  • Consumer actor IDs must be pseudonymized using HMAC-SHA256 with a per-merchant secret salt: HMAC-SHA256(user_id, per_merchant_salt). Never store raw user IDs in artifact payloads.
  • Email addresses, names, and physical addresses must not appear in chain-plane artifact payloads. The chain plane carries the address hash (§3.5, §3.7); the full delivery address lives in the raw plane per §8.2.1 and is disclosable at dispute time through the custodian disclosure flow.
  • Session tokens in discovery/referral payloads must be time-limited (maximum 24 hours), merchant-scoped, and verifiable — never raw session identifiers.
  • The entire artifact chain must be exportable without exposing any PII that could identify the consumer to a third-party reviewer.

§8.2 MUST-1: session_id_hash construction. session_id_hash MUST be HMAC-SHA256 with a per-merchant server-side salt, never the raw token. The salt rotation policy is defined in §2.7. The Referral payload MUST NOT carry a raw session_token (§3.2).

8.2.1 Two-Plane Evidence Model (Evidence Custodian) NEW in rc.2

AEP evidence is held in two planes:

  • Chain plane — the hash chain as defined throughout this specification: pseudonymized, hashed identifiers, exportable without exposing PII.
  • Raw plane — clear-text, dispute-usable identifier values: customer account/login ID, full delivery address, device ID (≥15 characters), and public IP address. The device fingerprint (≥20 characters) MAY remain hashed. Raw-plane values are held encrypted by an Evidence Custodian.

The Evidence Custodian is a role, not a product: any party operating under the obligations below may hold the raw plane. (A-Comm Vault is the reference implementation of the role.) The following custodian obligations are normative:

  1. Purpose limitation. Raw-plane values MAY be used only for dispute defense, fraud prevention, and regulatory or audit response. Any secondary use is prohibited.
  2. Segregated key material. Raw-plane encryption keys and the §2.7 hashing salts MUST be stored segregated from the chain plane.
  3. Trigger-gated access. Raw-plane values MUST be accessible only on a documented dispute trigger (dispute notification, pre-dispute inquiry, or regulatory demand).
  4. Sealed disclosure log. Every re-disclosure of raw-plane values MUST be recorded as a sealed disclosure-log artifact, so that the disclosure trail is itself tamper-evident.
  5. Erasure semantics. Erasure of a data subject's raw-plane values is performed by key-shred of the raw plane (GDPR Art. 17); the hash chain is retained under Art. 17(3)(e) (establishment, exercise, or defense of legal claims). The raw-plane retention clock MUST NOT exceed the chain retention clock (§8.5).

Rationale: card-network compelling-evidence programs require clear-text identifier values for element matching — hashed values do not qualify (§6.4). The two-plane model preserves AEP's PII-minimal chain while making the chain's evidence usable in those programs.

8.3 PCI-DSS Compliance

Authorization artifact payloads must never contain primary account numbers (PANs), CVVs, full magnetic stripe data, or any value that passes the Luhn algorithm.

Luhn check enforcement

Implementations must validate authorization payloads at ingestion using a Luhn check against all digit sequences of 13–19 digits. Any payload field containing a PAN-like pattern must be rejected before the artifact is created:

// PAN detection pattern
const PAN_REGEX = /\b(\d[ -]?){13,19}\b/g;

function passesLuhn(digits: string): boolean {
  const d = digits.replace(/\D/g, '');
  let sum = 0;
  let odd = false;
  for (let i = d.length - 1; i >= 0; i--) {
    let n = parseInt(d[i], 10);
    if (odd) { n *= 2; if (n > 9) n -= 9; }
    sum += n;
    odd = !odd;
  }
  return sum % 10 === 0;
}

function containsRawPaymentCredentials(payload: Record<string, unknown>): boolean {
  const text = JSON.stringify(payload);
  const matches = text.match(PAN_REGEX) ?? [];
  return matches.some(m => passesLuhn(m.replace(/\D/g, '')));
}

The only permitted payment credential field is payment_method_token — an opaque PSP-issued token that references a stored payment method without containing any card data. (The name payment_token_reference, used in earlier drafts, is an alias for payment_method_token for one MINOR cycle.)

Decoded-envelope gate NEW in rc.2

Pattern matching over the serialized payload is insufficient: base64 and JWT encodings defeat the PAN regex above. The ingestion gate MUST base64- and JWT-decode every envelope-shaped field (compact JWT/JWS/SD-JWT segments, and base64 blobs such as raw_protocol_payload) and run the PAN/Luhn check plus sensitive-authentication-data detection (security codes, track data) on the decoded claims before the artifact is created. Envelopes that would fail the gate are handled per §3.9.20: a SHA-256 digest plus extracted non-sensitive claims only.

8.4 Idempotency

Artifact creation must be idempotent. Implementations must derive idempotency keys deterministically from the artifact's identifying properties, allowing retries without duplicate artifact creation.

Event-native keys preferred (rc.2). Implementations SHOULD derive the idempotency key from an event-native identifier when one exists — the PSP event ID, the carrier event ID, or a protocol-supplied jti/nonce. The wall-clock construction below is a fallback for events with no native identifier only:

idempotency_key = SHA256(
  transaction_id + ":" +
  artifact_type  + ":" +
  actor_id       + ":" +
  floor(timestamp_ms / 60000)  // 60-second window
)

The idempotency key must be stored with a UNIQUE constraint on the artifacts table. A duplicate key violation on insert should return the existing artifact rather than raising an error.

Out-of-order ingestion (rc.2). Artifacts are ordered by seq, not by arrival: ingestion MAY be out of order pre-seal, and implementations MUST reorder artifacts into seq order before sealing. Post-seal appends (§2.5) arrive in any order.

8.5 Data Retention

Evidence chain data must be retained long enough to support dispute resolution, audit, and regulatory review. AEP specifies the following retention model:

Minimum retention Default retention Maximum retention Rationale
12 months 24 months 24 months Card network chargeback windows (120 days) + appeals (18 months) + SOC2 audit cycles

The retention_until timestamp is computed when the transaction is closed (completed_at is set): retention_until = completed_at + 730 days. Implementations should schedule automated deletion after this date.

Legal hold (NEW in rc.2). Automated deletion MUST be suspended for any chain referenced by an open dispute, arbitration, litigation, or regulatory inquiry, until the matter closes plus a defined tail (e.g., 90 days). The raw-plane retention clock (§8.2.1) MUST NOT exceed the chain retention clock.

8.6 Minimal Merchant Profile NEW in rc.2

A merchant MAY conform to AEP under a reduced profile consisting of the artifacts the merchant can emit from its own systems:

  • intent, cart, and authorization artifacts (plus post-seal fulfillment and refund), merchant-signed per §4.3;
  • protocol per §3.9 as observed, or GENERIC when no protocol envelope is observed.

Discovery, referral, and delegation artifacts are emitted by agent platforms and PSPs when available. Their absence MUST NOT penalize chain completeness for this profile: the expected artifact counts of §2.2 apply only to the Full profile. A Minimal-profile chain is chain_complete: true when its own expected artifacts are present and verify.

8.7 Minimal Platform Profile NEW in rc.2

An agent platform participates in AEP under this profile by doing either or both of the following:

  • countersigning the Intent artifact (§3.3) — product, price, quantity, and agent identity, signed with the platform's existing keys; and/or
  • supplying request_binding material (§3.9.19) for requests its agents originate.

A platform under this profile has NO obligation to emit query_category, citation_position, or any ranking-derived field. Chains carrying a platform-countersigned Intent artifact are eligible for elevated tiers per §3.9.12. The §6.5 named-actor rights and safe-harbor term apply to all platform-emitted artifacts.

CAIP-2 Chain Agnostic Improvement Proposal 2; canonical blockchain network identifier (e.g., eip155:84532, solana:5eykt4UsFv8P8NJdTREpY1vzqKqZKvdp). See §3.9.10.
Closed mandate A mandate signed at the moment of transaction confirmation against the specific final values (cart, merchant, total). Strongest user authorization. Recorded in AEP as ap2_closed_*_mandate_sdjwt (§3.9.7).
Cryptographic cart-payment binding Verification by hash equivalence between the cart-confirmation hash and the payment-mandate transaction ID, replacing v1.0.1's semantic items+amount matching. Recorded in AEP via §3.5 cart_payment_binding and validated per §3.9.11 rule 4.
Direct mode (AP2) A purchase flow in which the user is present at checkout and signs both Open and Closed mandate pairs (or just Closed pairs in v0.2 simplified flow). Recorded as ap2_mode = "direct".
Mastercard Verifiable Intent (VI) Mastercard's 3-layer SD-JWT / KB-SD-JWT credential framework for agent-mediated commerce, with 8 constraint families and explicit Immediate / Autonomous modes. Recorded in AEP as mastercard_verifiable_intent (§3.9.15).
Open mandate A mandate signed in advance, carrying constraints (allowed merchants, max amount, recurrence, etc.) but not bound to a specific final transaction. Recorded in AEP as ap2_open_*_mandate_sdjwt (§3.9.7).
RFC 7515 (JWS) JSON Web Signature.
RFC 8707 (Resource Indicator) OAuth 2.0 Resource Indicators; canonical resource URIs for OAuth-protected APIs. Used in §3.9.9 MCP server canonical URI.
RFC 8785 (JCS) JSON Canonicalization Scheme; the normative canonical-JSON serialization for AEP per §4.1.
RFC 9421 (HTTP Message Signatures) The signature envelope for TAP (§3.9.6), UCP (§3.9.8b), and Web Bot Auth (§3.9.18).
RFC 9530 (Content-Digest) Used by UCP to bind request body to signature. Carried in §3.9.8b rfc9421_signatures[].content_digest.
RFC 9901 (SD-JWT) Selective Disclosure JWT; the envelope class for AP2 and Mastercard VI mandate carriers.

Appendix C — PSP Dispute-Evidence Mapping

⚠
Field kinds and limits (corrected in rc.2) customer_communication, receipt, and shipping_documentation are file-upload evidence fields: they carry file IDs referencing uploaded documents (rendered PDFs), not text.
